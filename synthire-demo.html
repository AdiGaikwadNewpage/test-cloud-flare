Product

Three personas. One platform.

Synthire unifies the full hiring loop — from job posting to feedback — across recruiters, interviewers, and the AI engine that powers everything in between.

Recruiter

Owns the hiring pipeline. Posts jobs, reviews ranked candidates, and orchestrates interviews end-to-end.

  • Post jobs with weighted scoring criteria
  • Upload resumes (drag-drop, bulk)
  • Score & rank candidates by AI fit
  • Schedule interviews with auto emails
  • Full analytics on funnel + sources

Interviewer

Scoped access to assigned interviews. Conducts sessions with AI-generated questions and resume context.

  • View assigned interviews only
  • Conduct timed sessions in-app
  • Submit structured feedback + scores
  • Browse AI-suggested questions live
  • Side-by-side resume + JD viewer

AI Engine

The autonomous layer running entirely on Workers AI. Turns raw documents into structured, ranked signal without leaving Cloudflare's edge.

  • Parse resumes into structured JSON
  • Generate 1024-dim embeddings (BGE)
  • Cosine-similarity vector search
  • Weighted dimension scoring (LLM)
  • Auto-generate interview questions
Workflow

The full hiring loop, automated end-to-end.

From job creation to hire decision — every step is instrumented, cached, and processed asynchronously at the edge.

01

Create Job

Recruiter fills job title, description, and scoring weights (skills, experience, education, location). Weights must sum to 100%. Stored as a row in D1 with JSON scoring config. Optionally upload a JD PDF.

Next.js 15 — UI form Cloudflare Workers — Hono API D1 — jobs table R2 — JD PDF storage React Query — cache + mutation
02

Upload Resumes

Drag-drop PDF or DOCX files. Magic byte validation on the Worker (%PDF / PK\x03\x04) rejects misnamed files before any processing. Valid files are stored in R2 at resumes/{companyId}/{timestamp}.ext.

Next.js — ResumeBatchModal Cloudflare Workers — multipart handler R2 — raw file storage Magic bytes — PDF/DOCX validation
03

Text Extraction

Worker reads the R2 object and extracts plain text. DOCX uses mammoth (converts Word XML to text). PDF uses unpdf — a Workers-compatible wrapper around pdf.js that runs entirely at the edge without Node.js bindings. Both libraries run inside the Worker via nodejs_compat.

Cloudflare Workers — nodejs_compat mammoth — DOCX → text unpdf (pdf.js) — PDF → text R2 — fetch raw file bytes
04

LLM Resume Parsing

Extracted text is sent to Workers AI with a structured prompt. The LLM returns typed JSON: name, location, technical_skills[], experience[] (with from/to/stack[]), education[]. KV is checked first — cache hit skips the LLM entirely (TTL: 7 days). Fallback chain: Nemotron-120B → Llama-3.1-8B-AWQ.

Workers AI — native LLM binding @cf/nvidia/nemotron-3-120b — primary @cf/meta/llama-3.1-8b-instruct-awq — fallback KV — parse result cache (7 days) Neurons budget guard (daily limit)
05

Vector Embedding

The parsed candidate profile is serialised into a single text block and passed to Workers AI. The bge-large-en-v1.5 model returns a 1024-dimensional float vector. This vector is upserted into Vectorize with the candidate ID as metadata — enabling cosine similarity queries against job descriptions.

Workers AI — bge-large-en-v1.5 Vectorize — 1024-dim cosine index 1024-dim float32 vector Metadata: candidateId, jobId
06

Candidate Scoring

Two signals are combined into a final 0–100 score. Semantic (30%): cosine similarity between candidate vector and job description vector via Vectorize query. LLM dimension scores (70%): a second Workers AI call scores the candidate on each job dimension (skills, experience, education, achievements) × recruiter weights. Both scores are written to the candidates D1 row.

Vectorize — cosine similarity (30%) Workers AI LLM — dimension scores (70%) D1 — write overall_score + dimension_scores Formula: 0.3 × cos + 0.7 × weighted_llm
07

Async Background + Polling

The upload returns { candidateId } immediately (202). The pipeline runs in the background via ctx.waitUntil() — no long-lived HTTP connection needed. The frontend polls GET /api/candidates/:id every 2 seconds, reading processing_status to advance each file through parsing → scoring → complete.

Workers — ctx.waitUntil() background task Next.js — 2s polling interval States: parsing · scoring · complete · failed React Query — invalidate on complete
08

Pipeline Management

Scored candidates appear in the Kanban pipeline board. Recruiters drag cards between stages: New → Shortlisted → Scheduled → Technical → Culture fit → Hired / Rejected. Each drag fires a PATCH /api/candidates/:id with optimistic UI — the card moves instantly and rolls back on failure.

Next.js — Kanban drag-and-drop React Query — optimistic update + rollback Workers — PATCH /api/candidates/:id D1 — candidates.status column
09

Schedule Interview

Recruiter picks interview type (from configurable rounds), interviewer, date/time, and video link. The Worker inserts a row into interviews (D1) then calls queueEmail() which writes three rows to email_queue (D1): interviewer magic link, candidate confirmation, and a 24-hour reminder. All emails are sent asynchronously by cron — zero latency on the API response.

Workers — POST /api/interviews D1 — interviews + email_queue tables Cron Trigger — processEmailQueue every 1 min SendGrid — transactional delivery 3 emails queued per schedule action
10

Conduct Interview

Interviewer opens their unique link (from the magic-link email). The conduct page loads: candidate resume PDF streamed from R2 via authenticated Worker endpoint, JD PDF from R2, AI-generated questions (LLM prompt → KV cached 7 days), and the original feedback from previous rounds. A session timer starts on click.

R2 — stream resume + JD PDFs Workers — authenticated stream endpoint Workers AI — AI question generation KV — question cache (7 days) D1 — fetch prior feedback Next.js — in-browser PDF iframe
11

Feedback & Analytics

Interviewer submits structured scores (technical, communication, culture) + hire/no-hire recommendation. Written to interview_feedback (D1). Analytics queries aggregate across all candidates: funnel counts by stage, time-to-hire by month, activity feed (last 20 stage changes), and email delivery stats via SendGrid webhook.

Workers — POST /api/interviews/:id/feedback D1 — interview_feedback table D1 — analytics aggregation queries SendGrid webhook — delivery status (HMAC-SHA256) Next.js — analytics dashboard

Data Model

11 tables. Zero ORMs.

Prepared statements on D1 with JSON columns parsed in typed query helpers. Every JSON read is wrapped in try/catch.

companies

One row per workspace. Tenant root — every other row scopes to a company_id.

users

Recruiter, interviewer, or admin. Bcrypt-hashed passwords, scoped to a company.

jobs

Job postings with weighted scoring criteria. Weights stored as JSON and must sum to 100.

candidates

Parsed resume JSON + AI scores + processing status. R2 key + Vectorize ID stored here.

interview_types

Configurable rounds — Phone Screen, Technical, Culture Fit — defined per company.

interviews

Scheduled sessions linking candidate × interviewer × type, with email tracking columns.

interview_feedback

Per-interviewer structured scores and hire recommendation. Aggregated for analytics.

email_logs

SendGrid send log with delivery status updated via event webhook callbacks.

email_queue

Async send queue with atomic UPDATE…RETURNING dequeue, retry, and exponential backoff. status + claimed_at prevent double-send on concurrent cron runs.

refresh_tokens

Long-lived (30-day) opaque tokens stored SHA-256 hashed. Rotated on use. Enables silent re-auth without re-login — access tokens expire in 15 minutes.

Email System

Decoupled, durable, async by default.

No request ever blocks on email delivery. Inserts hit D1 in milliseconds; cron handles the rest.
